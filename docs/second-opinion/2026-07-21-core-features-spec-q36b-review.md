# Review: Core Features Implementation Specification (2026-07-21)

**Reviewed by:** Qwen 3.6B (Q36B)
**Review Date:** 2026-07-21
**Spec Status:** Draft
**Overall Assessment:** ⚠️ **Major gaps in multi-tenant isolation, missing cross-cutting concerns. Not ready for implementation.**

---

## Summary

This is a well-structured spec covering 7 core features (4.1.1–4.1.7) with clear data models, API endpoints, UI components, business logic, and acceptance criteria. The writing is consistent and the cross-references between sections are logical. However, there are **critical data model gaps** that would cause data leakage between tenants at runtime, **missing cross-cutting concerns** (error handling, rate limiting, pagination), and **no operational/infrastructure section** at all. These are not nitpicks — they are blockers.

---

## 🔴 Critical Issues (Block Implementation)

### 1. Missing `organization_id` on tenant-scoped tables

The spec establishes RLS (Row-Level Security) with `organization_id` on the `organizations` and `users` tables, but several child/related tables **omit `organization_id` entirely**, breaking the multi-tenant isolation pattern:

| Table | Missing Column | Impact |
|-------|---------------|--------|
| `candidate_skills` | `organization_id` | Skills from Tenant A could be queried by Tenant B via JOIN on `candidate_id` |
| `candidate_documents` | `organization_id` | Same risk — document metadata is tenant-scoped |
| `candidate_timeline` | `organization_id` | Timeline events leak across tenants |
| `requisition_skills` | `organization_id` | Skill requirements leak across tenants |
| `shortlist_candidates` | `organization_id` | Shortlist membership leaks across tenants |
| `interviews` | `organization_id` | Interview data leaks across tenants |
| `placements` | `organization_id` | Financial data leaks across tenants |
| `sourcing_lists` | ✅ Present | OK |
| `sourcing_list_candidates` | `organization_id` | List membership leaks |
| `outreach_templates` | ✅ Present | OK |
| `outreach_sequences` | ✅ Present | OK |
| `activities` | ✅ Present | OK |
| `client_portal_sessions` | ✅ Present | OK |
| `client_feedback` | `organization_id` | Feedback leaks across tenants |
| `search_log` | ✅ Present | OK |
| `report_definitions` | ✅ Present | OK |

**Fix:** Add `organization_id UUID NOT NULL` (FK → organizations.id) to every table that doesn't already have it. The RLS policy shown in 4.1.1 is a template — it must be applied to ALL tenant-scoped tables, not just `candidates`.

### 2. No error response format defined

Every API endpoint section lists methods and paths, but **no error response schema** is defined. Clients need to know:
- What HTTP status codes are returned (400 vs 404 vs 409 vs 422)
- The shape of error responses (e.g., `{ "error": { "code": "DUPLICATE_EMAIL", "message": "...", "details": {} } }`)
- Whether validation errors return all fields at once or one at a time

This is a cross-cutting concern that should be defined once in a shared section, not repeated per endpoint.

### 3. No pagination standard defined

Section 4.1.2 mentions "paginated" for candidate listing, but there is no standard for:
- Page-based vs cursor-based pagination
- Default/max page size
- How to express pagination in responses (`{ "data": [...], "meta": { "page": 1, "total": 150 } }`)
- Whether the client can request unlimited results

Without this, every endpoint will be implemented inconsistently.

## 🟠 High-Priority Issues (Should Fix Before GA)

### 4. Partial unique index syntax is ambiguous

> `Index: UNIQUE (organization_id, email) where email IS NOT NULL`

PostgreSQL requires explicit syntax for partial unique indexes:
```sql
CREATE UNIQUE INDEX idx_candidates_unique_email ON candidates(organization_id, email) WHERE email IS NOT NULL;
```
The spec should show the actual SQL, not a prose description, since this is an implementation spec.

### 5. `fee_amount` as a computed column is not natively supported in PostgreSQL

> `fee_amount | DECIMAL(10,2) | computed`

PostgreSQL does not support generated/computed columns in the same way as MySQL. Options:
- Use a `GENERATED ALWAYS AS` column (PostgreSQL 12+)
- Use a trigger
- Compute in application code and store

The spec should pick one and document the choice. Storing a computed value without a database-level constraint risks data inconsistency.

### 6. No rate limiting strategy

The spec defines authentication (JWT 24h, refresh 30d) but says nothing about:
- API rate limits (per-user, per-tenant, per-IP)
- Throttling for AI-powered endpoints (resume parsing, AI summary, AI outreach)
- DDoS protection for the client portal (magic link endpoint)

AI endpoints are particularly expensive and need explicit rate limits to prevent cost explosions.

### 7. Missing `created_by` on `client_portal_sessions`

The magic link is generated by a recruiter/admin. The table has `client_email` and `client_name` but no `created_by` (FK → users.id). Without this, you cannot audit who generated a link, when, or for which requisition — a compliance gap.

### 8. OTP flow has a race condition

> "OTP is single-use, expires after 10 minutes"

The spec says verify-otp is a separate endpoint from the magic link. Between clicking the magic link and entering the OTP, the session could expire or be invalidated. The flow should specify:
- What happens if the OTP expires during entry
- Whether the magic link TTL and OTP TTL are independent or coupled
- Whether a failed OTP attempt shows "link expired" or "wrong code" — these are different UX paths

### 9. No soft-delete mechanism defined for most entities

Section 4.1.2 mentions soft-delete for candidates (`status=archived`), but:
- Requisitions have no soft-delete (only `closed` status)
- Shortlists have no soft-delete
- Outreach templates have no soft-delete
- Sourcing lists have no soft-delete

If retention policies require data preservation, all entities need a consistent soft-delete pattern.

### 10. `candidate_timeline` event_type values are not enumerated

The spec lists auto-generated events ("profile created, resume uploaded, status changed, AI summary generated, document added") but the `event_type` column has no CHECK constraint. This should be:
```sql
CHECK (event_type IN ('profile_created', 'resume_uploaded', 'status_changed', 'ai_summary_generated', 'document_added', 'candidate_merged', 'candidate_archived'))
```

## 🟡 Medium-Priority Issues (Should Address Before GA)

### 11. No API versioning strategy beyond `/api/v1`

The spec uses `/api/v1/` prefixes but doesn't address:
- What happens when v2 breaks compatibility
- Deprecation policy for v1
- How clients discover the API version

### 12. Search endpoint uses GET with query params for complex filters

Section 4.1.7 shows:
```
GET /api/v1/search/candidates?q=react&skills=TypeScript,Node.js&location=SF
```

For 7+ filter parameters, GET URLs become unwieldy and may hit URL length limits. Consider:
- POST with a JSON body for complex searches
- Or at minimum document the parameter limit

### 13. Dashboard materialized views — refresh strategy is underspecified

> "Dashboard metrics are materialized views refreshed every 15 minutes"

Questions:
- Is this a cron job, a trigger-based refresh, or application-level?
- What happens during refresh — do users see stale data or no data?
- Which tables feed which materialized views?
- What if a refresh fails?

### 14. Report exports — async notification mechanism undefined

> "Report exports are generated asynchronously; user receives notification when ready"

How? WebSocket? Email? Polling endpoint? The spec mentions WebSocket for client feedback (4.1.6) but doesn't connect it to report notifications.

### 15. No data migration strategy

The spec defines the target schema but says nothing about:
- How to migrate from any existing data
- Backward compatibility during deployment
- Rollback strategy if a migration fails

### 16. `data_region` constraint is too restrictive

> `CHECK (IN ('us','uk','eu'))`

What about other regions? A CHECK constraint with a hardcoded list means schema migration is needed to add a region. Consider a separate `regions` lookup table or at minimum document that this is intentional for v1.

### 17. No audit log schema defined

Section 4.1.1 mentions "Audit log service (all user CRUD events)" as an integration point, but:
- No `audit_logs` table is defined
- No audit log schema/format is specified
- No retention policy for audit logs (different from data retention)

This is a compliance requirement (mentioned EU AI Act compliance in 4.1.4) and should be a first-class entity.

## 🟢 Low-Priority Issues (Nice to Have)

### 18. No UI state/error/empty states defined

UI components are described in their happy path, but:
- What does the candidate grid look like with 0 results?
- What error state does the Kanban show when a submission fails?
- Loading skeletons vs spinners?

### 19. No OpenAPI/Swagger documentation strategy

The API endpoints are well-structured and would benefit from machine-readable OpenAPI spec generation. Not critical for v1, but worth noting.

### 20. `kms_key_id` on organizations is a VARCHAR

This is fine for v1 (placeholder), but should be documented as "TBD — integration with AWS KMS / HashiCorp Vault" rather than left as a bare string column.

### 21. No health check endpoint

Standard practice: `GET /api/health` and `GET /api/ready` for load balancer health checks. Not mentioned anywhere.

### 22. `currency` default is 'USD' but no validation

The `requisitions.currency` column defaults to USD but has no CHECK constraint or reference to a currencies table. Non-ISO currency codes would silently corrupt fee calculations.

## Cross-Section Consistency Issues

### 23. Status enums are inconsistent

| Section | Entity | Status Values |
|---------|--------|--------------|
| 4.1.2 | Candidate | sourced, in_review, submitted, interviewing, placed, rejected, archived |
| 4.1.3 | Requisition | open, in_progress, submitted, filled, closed |
| 4.1.4 | Shortlist | draft, pending_approval, approved, submitted, rejected |
| 4.1.4 | Shortlist Candidate | pending, approved, rejected, withdrawn |
| 4.1.4 | Interview | advance, reject, hold |
| 4.1.4 | Placement | active, guarantee_period, completed, terminated |

Several of these overlap semantically (e.g., candidate "rejected" vs shortlist candidate "rejected" vs interview "reject"). Consider whether a shared status vocabulary would reduce confusion.

### 24. AI rank override compliance gap

Section 4.1.4 states: "If recruiter_rank != ai_rank, override_reason is mandatory (EU AI Act Art. 14 compliance)"

But section 4.1.2's candidate data model has no `ai_rank` column — AI rank only exists on `shortlist_candidates`. The spec should clarify that AI rank is **per-shortlist**, not a property of the candidate itself. This is already implied but should be explicit.

## Positive Observations

1. **Excellent cross-referencing** — Sections 4.1.2 through 4.1.7 reference each other cleanly (e.g., "AI Match (4.2.4) populates initial ai_rank")
2. **Acceptance criteria are testable** — Each section has 4-5 specific, verifiable acceptance criteria
3. **Business logic is well-reasoned** — Status transitions, fee calculations, and compliance requirements are thoughtfully specified
4. **UI components are descriptive** — Not just "a table" but "Card view with avatar, name, role, AI summary snippet, skill tags, source icon, date"
5. **The magic link + OTP flow for client portal** is a smart UX choice that balances security and friction

## Recommended Priority Order

1. **Fix all missing `organization_id` columns** (Issue #1) — This is a data leakage risk
2. **Define error response format and pagination standard** (Issues #2, #3) — Cross-cutting API concerns
3. **Add audit log schema** (Issue #17) — Compliance requirement
4. **Define rate limiting strategy** (Issue #4) — Cost control for AI endpoints
5. **Add infrastructure/deployment section** (Missing section) — Required for implementation
6. **Address all 🟠 issues** before code review
7. **Address 🟢 issues** during implementation sprint

---

## Verdict

**Status: RETURN TO AUTHOR** — The spec has strong structural quality and good feature coverage, but the missing `organization_id` columns on ~10 tables are a critical architectural flaw that would cause data leakage between tenants in production. This alone is sufficient to block implementation. The missing cross-cutting concerns (error format, pagination, rate limiting, audit logs) are also significant enough that implementing from this spec as-is would require substantial rework.

**Estimated rework effort if implemented as-is:** 2-3 sprints to fix data model gaps, add missing infrastructure, and define cross-cutting API conventions.
